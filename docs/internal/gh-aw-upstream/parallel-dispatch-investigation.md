# Parallel Dispatch Investigation: Can gh-aw Run Multiple repo-assist Instances Concurrently?

**Date:** 2026-03-02
**gh-aw version examined:** v0.51.4 (source at `/Users/skahessay/Documents/Projects/active/gh-aw`)
**Status:** Complete

## Executive Summary

Parallel dispatch of the same gh-aw workflow for different issues is **architecturally possible but requires changes to both our dispatch mechanism and our concurrency configuration**. The primary blocker is not gh-aw itself but our own concurrency group, which collapses all issue-triggered runs into a single slot. Repo-memory has a real but manageable race condition. GitHub API rate limits are not a practical concern at 3-5 concurrent runs. A safe parallelism cap of **3 concurrent runs** is recommended, with the option to increase to 5 after observing stability.

## 1. Concurrency Model

### How gh-aw Compiles `.md` to `.lock.yml`

The compilation pipeline is:

1. **`Compiler.CompileWorkflow(markdownPath)`** in `pkg/workflow/compiler.go` reads the `.md` file
2. **`ParseWorkflowFile`** extracts frontmatter (YAML between `---` fences) and markdown body
3. **`CompileWorkflowData`** validates and generates the GitHub Actions YAML
4. **`generateYAML`** orchestrates job generation via `buildJobs` in `compiler_jobs.go`
5. Output is written to `{name}.lock.yml` — a standard GitHub Actions workflow file

The lock file is a **complete, self-contained GitHub Actions workflow**. gh-aw does not add any runtime orchestration layer — once deployed, GitHub Actions is the sole executor.

### Concurrency Groups

gh-aw generates two levels of concurrency:

**Workflow-level concurrency** (top of the `.lock.yml`):
Generated by `GenerateConcurrencyConfig()` in `pkg/workflow/concurrency.go`. For issue-triggered workflows, the default pattern is:

```yaml
concurrency:
  group: "gh-aw-${{ github.workflow }}-${{ github.event.issue.number || github.event.pull_request.number || github.run_id }}"
  cancel-in-progress: true  # Only for PR workflows
```

For our `repo-assist.md`, the concurrency group is explicitly set in frontmatter:

```yaml
concurrency:
  group: "gh-aw-${{ github.workflow }}-${{ github.event.issue.number || github.event.pull_request.number || github.event_name }}"
  cancel-in-progress: true
```

**Engine-level (job) concurrency** (on the `agent` job):
Generated by `GenerateJobConcurrencyConfig()` in `pkg/workflow/concurrency.go`. For workflows with issue/PR/discussion triggers, **no default engine-level concurrency is applied** (the `hasSpecialTriggers()` function returns true, skipping it). This is correct — the workflow-level group handles it.

### Critical Finding: Concurrency Group Behavior

The concurrency group `gh-aw-${{ github.workflow }}-${{ github.event.issue.number || ... || github.event_name }}` has the following behavior:

| Trigger | Group resolves to | Effect |
|---------|------------------|--------|
| `issues:opened` for #100 | `gh-aw-Pipeline Repo Assist-100` | Unique per issue |
| `issues:opened` for #101 | `gh-aw-Pipeline Repo Assist-101` | Unique per issue |
| `workflow_dispatch` | `gh-aw-Pipeline Repo Assist-workflow_dispatch` | **All dispatches collapse** |

**This is why we currently have single-slot behavior.** Our `auto-dispatch.yml` triggers repo-assist via `gh workflow run repo-assist.lock.yml` (a `workflow_dispatch` event). All `workflow_dispatch` runs resolve to the same concurrency group `gh-aw-Pipeline Repo Assist-workflow_dispatch`, meaning each new dispatch cancels the previous one.

### Does gh-aw Support Multiple Simultaneous Runs?

**Yes**, but only when the concurrency group resolves to different values. If we dispatch repo-assist with issue context (so the group includes the issue number), multiple runs targeting different issues would run in parallel. The compiled `.lock.yml` is a standard GitHub Actions workflow — there is nothing in gh-aw that prevents parallel runs of the same workflow.

## 2. Repo-Memory Safety

### How Repo-Memory Is Implemented

Repo-memory is **entirely git-based** — no GitHub API gists, no database, no external storage. The implementation spans three layers:

1. **Clone phase** (`actions/setup/sh/clone_repo_memory_branch.sh`): At the start of the agent job, clones the `memory/{workflow-id}` branch to `/tmp/gh-aw/repo-memory/{id}/`. For `repo-assist`, this is branch `memory/repo-assist` cloned to `/tmp/gh-aw/repo-memory/default/`.

2. **Execution phase**: The AI agent reads/writes files freely in `/tmp/gh-aw/repo-memory/default/`. The prompt instructs it to use this directory for persistent state.

3. **Upload phase**: At the end of the agent job, the directory is uploaded as a GitHub Actions artifact (`repo-memory-default`).

4. **Push phase** (`actions/setup/js/push_repo_memory.cjs`): A separate `push_repo_memory` job downloads the artifact, checks out the memory branch, copies files, commits, and pushes. The push uses `git pull --no-rebase -X ours` (our changes win on conflict).

### Locking Mechanism

**There is no locking mechanism.** The source code (`push_repo_memory.cjs` and `clone_repo_memory_branch.sh`) contains zero references to locks, mutexes, atomics, or concurrency control. The sole conflict resolution is the git merge strategy:

```javascript
// From push_repo_memory.cjs line 376
execGitSync(["pull", "--no-rebase", "-X", "ours", repoUrl, branchName], { stdio: "inherit" });
```

### What Happens with Concurrent Read-Modify-Write

**Scenario:** Two repo-assist runs (A and B) start simultaneously, both targeting the same `memory/repo-assist` branch.

```
Time    Run A                          Run B
────    ─────                          ─────
T0      Clone memory/repo-assist       Clone memory/repo-assist
        (both get same snapshot)       (both get same snapshot)
T1      Agent modifies state.json      Agent modifies state.json
        (different changes)            (different changes)
T2      Upload artifact                Upload artifact
T3      push_repo_memory starts        push_repo_memory starts
T4      git add . && git commit        git add . && git commit
T5      git pull -X ours               git pull -X ours
T6      git push (succeeds)            git push (FAILS - remote changed)
T7      ---                            (no retry logic exists)
```

**The race condition is real.** Whichever run pushes second will fail its `git push` because the remote has advanced. The `-X ours` strategy only helps during `pull`, but if Run A pushed between Run B's pull and push, Run B's push fails.

Looking at the push script (`push_repo_memory.cjs` line 387):
```javascript
execGitSync(["push", repoUrl, `HEAD:${branchName}`], { stdio: "inherit" });
```

There is no retry-after-fail logic. A failed push calls `core.setFailed()` and the `push_repo_memory` job fails. **The agent's work is not lost** (the safe-outputs job handles PRs/comments independently), but the memory update is dropped.

### Severity Assessment

For our use case, repo-memory stores:
- Issues attempted (with outcomes)
- PRs created and their status
- A backlog cursor for round-robin processing
- Timestamps of last work

**Impact of lost memory update:** The next run re-reads stale memory and may re-attempt an already-attempted issue. The dedup checks in the workflow instructions (step 3b: check for existing `[Pipeline]` PRs) provide a safety net — the agent checks live GitHub state, not just memory. So duplicate work is caught at the PR-creation step, not the memory step.

**Verdict:** The race condition causes **inconvenience** (duplicate work starts then gets abandoned) but not **corruption** (no invalid state, no data loss beyond one run's memory update).

## 3. Rate Limits

### GitHub API Calls in a Typical Agentic Workflow Run

A repo-assist run makes GitHub API calls at several layers:

**Infrastructure calls (per run, fixed overhead):**
| Phase | Calls | Description |
|-------|-------|-------------|
| Activation job | ~5-10 | Team member check, timestamp validation, compute text |
| Checkout | 1 | `actions/checkout` (git clone, not API) |
| Repo-memory clone | 1 | git clone (not REST API) |
| Repo-memory push | 2-3 | git fetch + git push (not REST API) |
| Artifact upload/download | 2-4 | GitHub Actions artifact API |
| Safe-outputs processing | 5-30 | PR creation, comments, labels, issue updates |
| Rate limit check | 1-2 | `listWorkflowRuns` + `getCollaboratorPermissionLevel` |

**Agent execution calls (variable, depends on task):**
The Copilot engine uses the GitHub MCP server, which proxies API calls through a sandboxed gateway. A typical repo-assist run includes:
- `gh issue list`, `gh pr list`: ~5-10 calls (list API)
- `gh issue view`, `gh pr view`: ~5-15 calls (get API)
- `gh run list`, `gh run view`: ~3-5 calls (actions API)
- `gh pr diff`, `gh pr checks`: ~2-5 calls (get API)
- `gh workflow run`: ~1-2 calls (dispatch API)

**Estimated total per run: 50-100 GitHub API calls**

This is a rough estimate. The actual number depends heavily on how many issues the agent processes, how many PRs it maintains, and how many log-fetching operations it performs. A "busy" run (implementing 2-3 issues) could hit 150+ calls. An idle run (no work to do) might use only 30.

### Budget for 3-5 Concurrent Runs

GitHub's authenticated rate limit: **5,000 requests/hour** per token.

| Concurrent runs | Est. calls/hour (assuming 1 run/hour each) | % of budget |
|----------------|---------------------------------------------|-------------|
| 1 | 50-100 | 1-2% |
| 3 | 150-300 | 3-6% |
| 5 | 250-500 | 5-10% |

**Rate limits are not a concern.** Even at 5 concurrent runs with worst-case call counts, we use ~10% of the hourly budget. The theoretical maximum (5 runs x 150 calls each = 750) is well within the 5,000/hour limit.

Note: All concurrent runs use the same `GH_AW_GITHUB_TOKEN` (PAT). If we were using `GITHUB_TOKEN` (installation token), the limit would be 1,000 requests/hour per repository, which would be tighter (750/1000 = 75% at worst case). Our PAT-based setup avoids this.

### GitHub Actions Concurrent Job Limits

More relevant than API rate limits:
- **Free tier:** 20 concurrent jobs
- **Pro/Team:** 40 concurrent jobs
- **Enterprise:** 180 concurrent jobs

Each repo-assist run uses 4-6 jobs (pre_activation, activation, stop_time_check, agent, safe_outputs, push_repo_memory). At 3 concurrent runs, that is 12-18 jobs, which fits within the free tier. At 5 concurrent runs, 20-30 jobs — requires Pro tier.

## 4. Recommendation

### Is It Safe to Parallelize?

**Yes, with a cap of 3 concurrent runs.** The risks are manageable:

| Risk | Severity | Mitigation |
|------|----------|------------|
| Repo-memory race condition | Medium | Accept last-writer-wins; dedup checks in agent instructions prevent duplicate PRs |
| Concurrency group collapse | Blocker | Must change dispatch mechanism (see below) |
| Rate limits | Low | Well within budget at 3-5 runs |
| Runner job limits | Low | 3 runs = ~18 jobs, fits free tier |
| Git branch conflicts (feature branches) | Low | Each issue gets its own branch (`repo-assist/issue-N-desc`) |

### Required Changes to Dispatch Mechanism

**Change 1: Pass issue context to workflow_dispatch**

The repo-assist workflow needs a `workflow_dispatch` input for the issue number:

```yaml
# In repo-assist.md frontmatter
on:
  workflow_dispatch:
    inputs:
      issue_number:
        description: "Issue number to process (optional, for targeted dispatch)"
        required: false
        type: string
```

And the concurrency group needs to use it:

```yaml
concurrency:
  group: "gh-aw-${{ github.workflow }}-${{ inputs.issue_number || github.event.issue.number || github.event.pull_request.number || github.run_id }}"
```

**Problem:** gh-aw compiles the `.lock.yml` from the `.md` frontmatter. Custom `workflow_dispatch` inputs may or may not be supported by the gh-aw compiler. The `concurrency` field IS user-configurable (our `repo-assist.md` already overrides it). But `workflow_dispatch` inputs need investigation — gh-aw may strip or transform them during compilation.

**Change 2: Update auto-dispatch.yml to pass issue number**

```bash
# Current (single-slot)
gh workflow run repo-assist.lock.yml --repo "$REPO"

# Parallel-capable
gh workflow run repo-assist.lock.yml --repo "$REPO" -f issue_number="$ISSUE_NUMBER"
```

**Change 3: Remove the single-slot guard**

The current `auto-dispatch.yml` has an explicit guard (lines 77-98) that skips dispatch when repo-assist is already running:

```bash
ACTIVE_RUNS=$(gh run list --workflow repo-assist.lock.yml ...)
ACTIVE_COUNT=$(... | jq 'length')
if [ "$ACTIVE_COUNT" -gt 0 ]; then
  echo "skip=true"  # <-- This enforces single-slot
fi
```

Replace with a cap-based guard:

```bash
MAX_CONCURRENT=3
if [ "$ACTIVE_COUNT" -ge "$MAX_CONCURRENT" ]; then
  echo "skip=true"
  echo "repo-assist at capacity ($ACTIVE_COUNT/$MAX_CONCURRENT); deferring."
fi
```

**Change 4: Modify agent instructions for targeted mode**

When dispatched with an `issue_number` input, the agent should prioritize that issue rather than doing round-robin backlog processing. Add to the repo-assist.md prompt:

```markdown
## Targeted Dispatch Mode

If `${{ inputs.issue_number }}` is set and non-empty, this is a targeted dispatch.
Focus exclusively on implementing issue #${{ inputs.issue_number }}.
Skip the round-robin backlog scan and status update.
```

### What Remains Unknown

1. **gh-aw support for `workflow_dispatch` inputs in frontmatter.** The compiled lock file for repo-assist currently has `workflow_dispatch: null`. We need to verify whether the gh-aw compiler passes through custom inputs or strips them. If it strips them, we need a different approach (e.g., using `repository_dispatch` with a client payload instead).

2. **Whether the Copilot engine has internal concurrency awareness.** The engine runs inside a Docker container with sandboxed network access. If multiple engines run simultaneously and both try to create PRs for the same issue, the safe-outputs layer handles deduplication — but we have not tested this at scale.

3. **Repo-memory push retry behavior under contention.** The current implementation has no retry logic. Adding retry-with-backoff to `push_repo_memory.cjs` would significantly reduce the memory-loss risk, but that is a gh-aw upstream change.

### Recommended Implementation Order

1. **Phase 1 (no gh-aw changes needed):** Change `auto-dispatch.yml` guard from `ACTIVE_COUNT > 0` to `ACTIVE_COUNT >= 3`. This allows multiple dispatches but they still collapse into the same concurrency group. Net effect: the last dispatch wins (others get cancelled). This is marginally better than today for burst scenarios.

2. **Phase 2 (concurrency group change):** Modify `repo-assist.md` concurrency group to include `github.run_id` as the unique key instead of `github.event_name`. This prevents cancellation of parallel runs. Requires testing that the compiled `.lock.yml` reflects the change correctly.

3. **Phase 3 (targeted dispatch):** Add `workflow_dispatch` input for `issue_number`, modify `auto-dispatch.yml` to pass it, and add targeted-mode instructions to the prompt. This is the full parallel dispatch solution.

4. **Phase 4 (repo-memory hardening):** Either add retry logic to our push step (local workaround) or file an upstream issue for retry support in `push_repo_memory.cjs`.
